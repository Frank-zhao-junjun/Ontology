# 配置导出器草案

## 目标

定义建模工具点击“导出”后，如何从冻结的项目版本生成一套可加载的配置包。该草案面向第一版实现，强调稳定性、一致性与可扩展性，而不是一次性生成所有复杂能力。

## 设计原则

1. 导出输入必须是已冻结的 ProjectVersion，而不是编辑中的临时状态。
2. 导出器只消费标准化后的四大元模型：数据、行为、规则、事件。
3. 配置包中的数据结构共享同一份 ExportContext，避免字段、状态机定义重复推导。
4. 第一版优先保证配置包可加载、可扩展、可追踪，不追求一次生成所有业务细节。
5. 每次导出都产出 manifest.json，便于版本加载、问题追踪和回放。

## 核心接口

```ts
export interface ExportConfig {
  includeData: boolean;
}

export interface ExportedFile {
  path: string;
  content: string;
}

export interface ConfigPackageManifest {
  projectId: string;
  version: string;
  generatedAt: string;
  entityCount: number;
}

export interface ConfigPackage {
  files: ExportedFile[];
  manifest: ConfigPackageManifest;
}

export interface ExportContext {
  projectVersion: ProjectVersion;
  exportConfig: ExportConfig;
  metamodels: NormalizedMetamodels;
  entities: NormalizedEntity[];
  aggregateRoots: NormalizedEntity[];
}

export interface Exporter {
  export(projectVersion: ProjectVersion, exportConfig: ExportConfig): Promise<ConfigPackage>;
}
```

## 标准化模型

```ts
export interface NormalizedMetamodels {
  data: DataModel;
  behavior: BehaviorModel;
  rules: RuleModel;
  events: EventModel;
}

export interface NormalizedEntity {
  id: string;
  name: string;
  nameEn: string;
  fileName: string;
  className: string;
  tableName: string;
  isAggregateRoot: boolean;
  attributes: NormalizedAttribute[];
  relations: NormalizedRelation[];
  stateMachine?: StateMachine;
  rules: Rule[];
  events: EventDefinition[];
}
```

## 导出流程

```ts
export class ConfigExporter implements Exporter {
  async export(projectVersion: ProjectVersion, exportConfig: ExportConfig): Promise<ConfigPackage> {
    validateProjectVersion(projectVersion);

    const metamodels = normalizeMetamodels(projectVersion.metamodels);
    const entities = normalizeEntities(metamodels);

    const context: ExportContext = {
      projectVersion,
      exportConfig,
      metamodels,
      entities,
      aggregateRoots: entities.filter(entity => entity.isAggregateRoot)
    };

    const files = [
      ...emitRootFiles(context),
      ...emitDataFiles(context)
    ];

    return {
      files,
      manifest: buildManifest(context, files)
    };
  }
}
```

## 第一版导出范围

### Root

- config.json
- README.md
- manifest.json

### Data

- data/entities.json
- data/state_machines.json
- data/rules.json
- data/events.json
- data/seed_data.json (可选)

## 导出前校验

1. 版本号不能为空，且必须符合语义化版本格式。
2. 至少存在一个实体。
3. 实体 `nameEn`、属性 `nameEn`、关系定义必须可生成稳定标识。
4. 状态机、规则、事件引用的实体 ID 必须存在。
5. `includeData = false` 时禁止生成示例数据内容。

## 可测试输出契约（JSON Schema 最小草案）

以下 Schema 用于自动化验收“导出结果可观测、可复现、可判定”。

### manifest.json

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["projectId", "version", "generatedAt", "entityCount"],
  "properties": {
    "projectId": { "type": "string", "minLength": 1 },
    "version": { "type": "string", "pattern": "^\\d+\\.\\d+\\.\\d+$" },
    "generatedAt": { "type": "string", "format": "date-time" },
    "entityCount": { "type": "integer", "minimum": 1 }
  },
  "additionalProperties": false
}
```

### config.json

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["project", "version", "generatedAt"],
  "properties": {
    "project": { "type": "string", "minLength": 1 },
    "version": { "type": "string", "pattern": "^\\d+\\.\\d+\\.\\d+$" },
    "description": { "type": "string" },
    "generatedAt": { "type": "string", "format": "date-time" }
  },
  "additionalProperties": true
}
```

### data/entities.json

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "array",
  "minItems": 1,
  "items": {
    "type": "object",
    "required": ["id", "name", "nameEn", "isAggregateRoot", "attributes", "relations"],
    "properties": {
      "id": { "type": "string", "minLength": 1 },
      "name": { "type": "string", "minLength": 1 },
      "nameEn": { "type": "string", "minLength": 1 },
      "isAggregateRoot": { "type": "boolean" },
      "attributes": { "type": "array" },
      "relations": { "type": "array" }
    },
    "additionalProperties": true
  }
}
```

测试建议：
1. 导出接口测试后，自动解压配置包并逐个执行 Schema 校验。
2. Schema 校验失败视为导出失败，不允许进入运行时加载测试。
3. `includeData=false` 时，断言 `data/seed_data.json` 不存在。

## 模板策略

1. 固定骨架文件使用模板输出，不在代码里直接拼接长字符串。
2. 复杂内容只传结构化上下文给模板，例如 `entities`、`stateMachines`、`rules`。
3. 配置文件模板统一维护，确保格式一致性。

## 第一版非目标

1. 不在第一版中生成完整测试用例。
2. 不在第一版中支持远程服务器自动部署。
3. 不在第一版中支持多数据库方言。
4. 不将运行时 AI 能力建模成独立“流程模型”，而是由运行时系统根据四大元模型动态构建上下文。

## 实施顺序建议

1. 先实现 `types.ts`、`validators`、`normalizers`。
2. 再实现 `data` emitter，优先保证配置数据可加载。
3. 然后补 `root` emitter，生成配置包的根文件。
4. 最后补 `packagers` 与导出接口整合。
