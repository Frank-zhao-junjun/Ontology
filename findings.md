# EPC页签设计发现

- EPC规格说明书要求输出事件、功能、规则、组织单元、信息对象、流程链、矩阵、异常、KPI、系统集成、合规要求、自检等完整章节。
- 当前仓库已有四大元模型：数据、行为、规则、事件；前台已不暴露旧流程模型。
- 当前processModel是AI编排步骤模型，不适合作为EPC业务流程真值模型。
- EPC skill可复用的核心价值是模板结构、命名规范、完整性检查项与企业级文档框架。
- 现有四大元模型无法单独完整推出组织单元、执行系统、输入输出、SLA、KPI、合规项，需要补充EPC专用元数据层。
- EPC正式类型定义应以 `EpcModel -> EpcAggregateProfile` 为中心，并与 `OntologyProject` / `ProjectVersion.metamodels` 形成弱耦合集成。
- 为避免打断现有可运行状态，`epcModel` 与版本快照中的 `epc` 字段先按可选字段接入。
- 第一阶段骨架已验证可行：聚合根详情可显示 EPC 页签，store 能自动初始化/再生成 profile，预览可输出 Markdown 规格说明草案。
- 第一阶段仍是“文档级补充字段”模型，尚未开放组织单元、系统、信息对象、连接器、KPI、合规项的精细编辑。
- EPC信息对象的正确边界是“流程视角下的信息载体”，不是新的数据模型、元数据或主数据真值层。
- 第二阶段信息对象编辑规则应遵循：派生对象结构只读、手工对象可增删改、重生成刷新派生对象并保留手工对象与流程说明。
- 组织单元与执行系统同样属于 EPC 补充元数据，不应从四大元模型臆造；当前实现改为手工维护并纳入 Markdown 与完整性校验。
- EPC 导出已按设计方案接入 config package：导出 `data/epc.json` 结构化快照、`epc/{aggregate}.json`、`epc/{aggregate}.md`，并在 manifest 中追加 `epcCount`、`epcAggregates`、`generatedEpcAt`。
- EPC 页签级下载入口可直接复用配置包中的 EPC 文件形态：页签内直接下载 `{aggregate}.md` 与 `{aggregate}.json`，避免用户必须先走整包导出。
- EPC 页签整包下载入口直接复用 `/api/export`，这样页签下载与系统导出结果完全同构，不需要在前端维护额外打包逻辑。

## 2026-04-21 US / TDD 复核发现

- 当前仓库测试强项集中在：导出链路、主数据动态表、属性主数据引用、元数据模板绑定、聚合角色、EPC 生成与导出、运行时版本切换。
- 当前仓库测试弱项集中在：项目分组展示与删除策略、业务场景 CRUD 与删除保护、状态机深层规则、规则高级能力、事件订阅高级能力、版本历史、AI 建议应用。
- `US.md` 的故事拆解是完整的，但“故事清单”和“实际测试证据”没有一一绑定，导致容易把“有实现”误判为“已验证”。
- `tests` 目录里此前没有直接覆盖 US-2.3 的工作台场景过滤测试；该能力已存在于 `modeling-workspace.tsx`，适合先补集成测试锁定行为。
- `tests/integration/runtime-version.spec.ts` 暴露出一个重要信号：版本切换能力虽然有测试，但断言里仍保留“manifest.version 固定为 1.0.0”的兼容注释，说明相关 US 只能算“部分验证”，不能算完全收口。
- US-1.1 的 store 实现此前缺少与 UI 一致的删除保护：界面会阻止删除有实体的项目分组，但 `deleteEntityProject` 本身仍允许无条件删除；直接单测补上后已暴露并修复这一断层。
- US-1.2 的工作台此前存在一个隐式错误：在“全部项目”视角下仍允许创建业务场景，并会把新场景静默分配给首个项目；直接集成测试已暴露并修复这一问题。
- `US.md` 原先仅写了“场景数量限制”而没有数字；本轮已由用户明确为“每项目最多 10 个业务场景”，并已同步到测试、store、工作台提示和 `US.md`。
- US-1.2 的 UI CRUD 原先并不完整：业务场景列表只有创建与选择，没有编辑/删除入口；本轮补齐后，工作台层终于具备与 store 一致的场景 CRUD 行为。
- US-2.1 的 store 之前只验证业务场景归属，不验证聚合角色边界；因此非法子实体、非法聚合根和“删根不删子”的问题都能直接穿透到状态层。本轮单测已暴露并修复这一断层。
- US-2.1 的“只有聚合根可以定义领域事件”当前主要由 `src/components/ontology/event-model-editor.tsx` 在 UI 层守卫，而不是由 store 二次校验；本轮已用直接集成测试锁定现状，后续若允许非 UI 路径写事件，还需要决定是否把这条约束下沉到状态层。
- US-2.1 的“聚合根变更处理”至少包含一条必须的 store 语义：仍被子实体引用的聚合根不能直接降级为 `child_entity`，否则会留下“子实体挂子实体”的非法链路；本轮单测已暴露并修复这一断层。
- US-2.1 当前已经形成一个明确、可执行的嵌套边界：`child_entity` 的父节点必须是 `aggregate_root`，因此模型深度被限制为“聚合根 + 一层子实体”；这条限制现在已有直接单测证据。
- US-1.3 的业务场景列表此前虽然在行内显示了实体数量，但列表本身没有按项目过滤，也没有搜索能力；这意味着“查看项目下业务场景列表”的核心验收实际上并未闭环。本轮已用直接集成测试补齐并修正该断层。
- US-4.1 当前的“状态码”概念在实现层等价于 `state.id`，因为类型与编辑器还没有单独暴露状态编码字段；本轮测试和 store 保护均按这一现状锁定，后续若产品要求“编码/名称分离”，需要先扩展 `State` 结构。
- US-4.1 之前最大的实现偏差不是“缺少状态 UI”，而是缺少状态机入口级约束：store 可以接受重复状态编码、多个初始态，也允许 UI 通过“删状态并同步删转换”绕过删除保护；本轮已在 store 层统一堵住。
- `src/components/ontology/behavior-model-editor.tsx` 之前删除状态时会静默过滤所有关联转换，这与 `TC-4.1.3 删除状态 - 有转换规则引用 - 失败` 相冲突；现已改为明确提示先移除转换规则。
- US-4.1 的“状态数量限制”现已由用户明确为“每个状态机最多 10 个状态”；当前 store 与行为编辑器都已按该阈值生效。
- US-4.1 现在已经具备 UI 层直接证据：行为编辑器里新增状态、重复编码拦截、第二个初始态拦截、被转换引用状态删除拦截都已有真实交互测试，不再只是 store 单测结论。
- 经过本轮决策补齐后，US-4.1 已不存在已知实现或测试缺口；后续若要继续扩展，只剩“状态编码/名称分离”这类产品增强项，而不是当前故事的验收缺口。
- US-6.1 之前虽然已有事件相关实现和 EPC/Store 间接覆盖，但缺少直接锁定“过去式命名、事务边界、状态变更条件”的测试；如果只看旧测试，很容易把“事件能保存”误判为“领域事件验收已闭环”。
- US-6.1 现在已经把事件约束从 UI 守卫下沉到了 store：即使绕过编辑器直接调用 `addEventDefinition` / `updateEventDefinition`，非聚合根事件、非过去式命名和缺少条件的状态变更事件也都会被统一拒绝。
- 默认事件创建语义已经稳定：编辑器默认生成 `实体已创建/更新/删除/变更` 命名，默认事务阶段为 `AFTER_COMMIT`，默认最小 payload 为 `id`；这让 EPC 和导出链路读取到的事件快照更接近 DDD 语义，而不是占位数据。
- US-4.2 之前最大的断层不是“没有转换 UI”，而是转换规则完全没有入口级校验：store 接受引用不存在状态的转换，编辑器也允许空状态和无条件自动转换直接落库。
- US-4.2 里“转换条件表达式”和“转换动作配置”在当前模型里分别落在 `preConditions` 与 `postActions`；只要这两个字段不进入直接测试，就很容易在后续重构中被无意删空而不自知。
- US-4.2 现已补齐独立的“编辑转换”交互入口，并有真实集成测试锁定“编辑 + 删除”路径；结合此前已补上的创建、完整性与条件校验，当前故事已达到“核心已验证”。
- US-1.1 此前缺的不是 store 基础 CRUD，而是工作台层的直接证据：如果没有真实集成测试，项目分组的实体计数、项目切换过滤和“删除当前选中项目后回退到全部项目”这些交互语义都只能靠肉眼判断。
- US-1.1 的 API 删除策略此前也只有成功路径；补上 DELETE 失败路径后，`/api/projects/[id]` 现在至少明确了“数据库删除异常 -> 500”这一条服务端契约，避免把“能删成功一次”误判成“删除策略已验证”。
- US-6.2 之前的类型定义虽然已经有 `retryPolicy`、`handlerId` 和 `idempotencyKeyPattern`，但如果没有 store 级校验与编辑器直接输入，这些字段只是“结构存在”，不能算“订阅能力已验收”；本轮补齐后，异步重试和同步幂等才真正闭环。
- 事件订阅这类规则更适合下沉到 store，而不是只留在事件编辑器里做表单判断；否则任何绕过 UI 的写入路径都可能把缺少重试策略、引用无效事件或空动作引用的订阅直接落库。
- `MetadataManager` 暴露出一个共性风险：对话框表单如果只渲染视觉上的 `Label`，却不显式绑定 `htmlFor` / `id` 或 `aria-labelledby`，那么测试选择器和真实可访问性会一起失真；这类表单在后续新增时应默认按可访问控件来写。
- US-4.3 的触发器模型现在不只是“能配置 trigger 字段”：store 已把 `eventId` / `cron` / `publishEventId` / `executionLogs` 作为统一约束与归一化入口，只有这样手动、事件、定时三类触发器的行为才能在 UI 外也保持一致。
- 当 Windows 终端里整条 `ci:check` 输出过长时，最稳妥的办法不是猜测交互卡住，而是按 `lint -> ts-check -> unit -> integration -> e2e smoke` 分段重跑；这样既能快速定位真实失败阶段，也能给故事收口留下清晰证据。
